§ 01 — Core Principle

Fetch once, run local

The game downloads everything it needs on first install, then runs entirely from the player's device. The server is contacted only to deliver the initial content bundle and to check for updates. No gameplay action requires a live internet connection.

◎

The client owns all runtime behaviour after bootstrap

Game logic, state, player progress, and all decisions happen on-device. The backend is a content delivery and data collection service — nothing more. This is the constraint that shapes every decision in ADR 001 and ADR 002.

§ 03 — Frontend Modules (ADR 001)

What runs on the player's device

Every component below runs entirely on the player's phone or tablet after the first install. Nothing here requires a live server connection during gameplay.

Client Domain — Frontend

ADR 001 · PROPOSED
PWA Shell (SvelteKit)

The installable app container. SvelteKit provides the project structure, routing, and page rendering. Svelte compiles to plain JavaScript — no framework runtime is bundled into the final download, keeping the install size small.

Build Toolchain — Vite + vite-plugin-pwa

The developer tool that compiles source code into the final app. vite-plugin-pwa automatically generates the Service Worker and Web App Manifest, removing the most complex parts of PWA setup from the developer's plate.

Service Worker

A background script that intercepts network requests and serves cached content when the player is offline. After the first install, the app loads instantly from this cache — no internet needed. Also detects when a new content bundle is available via IF-4 and triggers a re-download.

Web App Manifest

A configuration file (auto-generated by vite-plugin-pwa) that tells the browser how to display and install the app — its name, icon, theme colour, and launch behaviour. This is what enables "Add to Home Screen" without an app store.

IndexedDB via Dexie.js

The on-device database. Stores the complete game bundle received from IF-1 (all locations, quests, items, events) and all player progress generated during gameplay. Dexie.js provides a clean interface over raw IndexedDB and manages schema versioning — every future change to what is stored must be written as an explicit, numbered migration to avoid data loss.

State Manager (Svelte stores)

Holds the current runtime state of the game in memory — what the player can see and interact with right now. Reads from IndexedDB on launch, writes back on change. The UI reacts to state changes automatically through Svelte's reactivity system.

Location Engine

Reads the device's GPS coordinates and checks them against the location data stored in IndexedDB. When the player enters a defined zone, the Location Engine fires a trigger into the Quest/Event FSM.

Quest / Event FSM (Finite State Machine)

The core game logic module. Manages the lifecycle of every quest and event — which states are possible (available, active, complete, failed), and what transitions are allowed between them. All game decisions run through this module. Writes results back to the State Manager and IndexedDB.

Scoring Module

Calculates end-of-game scores based on completed quests, visited locations, and other criteria defined in the game content bundle. Reads from the State Manager; output is displayed in the UI.

Analytics Queue

Collects gameplay events (location visited, quest completed, session started) and sends them to the server via IF-3 when a network connection is available. Fires independently from gameplay — if the network is down, events are queued or dropped silently. The game never pauses waiting for this.

Version Check (IF-4 client side)

Runs automatically on every app launch before the game loads. Sends the currently stored bundle_version to the server. If the server says a new version is available, the Service Worker triggers a fresh download via IF-1. If the server is unreachable, the game launches normally from the local bundle.

§ 04 — Backend Modules (ADR 002)

What runs on the server

The entire backend is a single PocketBase binary running on one VPS. It covers all four interfaces. It does not run game logic.

Backend Domain — PocketBase on VPS

ADR 002 · PROPOSED
Nginx — Reverse Proxy + HTTPS

Sits in front of PocketBase and handles all incoming traffic. Terminates HTTPS connections (using a TLS certificate from Certbot/Let's Encrypt) and forwards requests to PocketBase. HTTPS is not optional — service workers will not function without it, which means this must be configured before any frontend live testing can begin.

PocketBase — REST API Layer

The public-facing API that the frontend calls. Handles all four interfaces: serving the bootstrap bundle (IF-1), receiving analytics events (IF-3), and responding to version checks (IF-4). Stateless — each request is self-contained with no session memory between calls.

PocketBase — Admin UI / CMS (IF-2)

A browser-based control panel built into PocketBase. The game designer and data dev use this to create and update game content — locations, quests, items, events — without writing code. This is the CMS for the project. The backend dev configures the data structure (collections); the content team fills it in.

SQLite Database

The database PocketBase uses to store everything — game content authored through the Admin UI, analytics events received from the client, and the current bundle version identifier. Stored as a single file on the VPS. Simple to back up; does not support multiple simultaneous writers or multiple machines.

File Storage

PocketBase's built-in file management, used to store static bundle assets (images, audio, or other media) that accompany the JSON game content. The frontend downloads these as part of or alongside the IF-1 bootstrap bundle.

Auth & Security (available, not yet scoped)

PocketBase includes built-in authentication and access control. Not required for the demo (the game has no player login), but available if player accounts or protected content authoring become a requirement. No additional setup is needed to unlock this — it is part of PocketBase out of the box.

§ 05 — Interfaces (IF-1 through IF-4)

The only permitted cross-domain connections

These four interfaces are the only points where the frontend and backend are allowed to communicate. Everything else is internal to one domain. The contracts below must be agreed before implementation begins — changes after the fact require sign-off from all touching roles.

Interface Direction What crosses the boundary Owner Frontend responsibility Backend responsibility
IF-1
Bootstrap Endpoint		 Backend → Client Versioned JSON bundle: all locations, quests, items, events. Includes a bundle_version hash. Backend Dev Data Dev Frontend Dev Request bundle on first install. Store in IndexedDB via Dexie. Show install progress to player. Handle failed downloads gracefully. Expose the endpoint. Serve the correctly formatted bundle. Ensure the bundle_version field is present and accurate.
IF-2
Content Schema		 Data Platform → Backend (internal) The structure of game content in PocketBase collections and how it maps to the IF-1 bundle format. Not a public API — invisible to the frontend. Backend Dev Data Dev Does not implement this. Depends on its output: the IF-1 bundle shape must be stable before the frontend data layer is built. Configure PocketBase collections to match the agreed schema. Build the transformation that produces the IF-1 bundle from raw CMS content.
IF-3
Analytics Events		 Client → Backend Structured event payloads (session start, location visited, quest completed, etc.). Optional — gameplay is never affected by whether this succeeds. Frontend Dev Backend Dev Analysts Instrument the Analytics Queue to fire events at the correct moments. Ensure events are never blocking — fire-and-forget only. Queue unsent events if offline. Expose the ingest endpoint. Accept payloads and store them in SQLite. Pass data to the analytics pipeline (ADR 003).
IF-4
Version Check		 Client → Backend → Client A lightweight ping: client sends its stored bundle_version; server replies "up to date" or "new version available." Frontend Dev Backend Dev Send the version check on every launch before loading the game. If the server is unreachable, launch anyway from the local bundle. If a new version is available, trigger an IF-1 re-download before play begins. Expose a fast, lightweight endpoint. Compare the submitted bundle_version against the current value and return the correct response.
§ 06 — Proposed Stack Summary

Technology choices at a glance

All choices below are proposed and subject to team review. Framework choice is open to revision based on the assigned frontend developer's existing experience.

SvelteKit
Frontend framework · PWA shell
Vite + vite-plugin-pwa
Build toolchain · service worker + manifest generation
Dexie.js (IndexedDB)
On-device database · game bundle + player progress storage
Svelte Stores
Reactive state management · in-memory game state
PocketBase
Single-binary backend · REST API + Admin UI + SQLite + File Storage
SQLite
Embedded database · no separate server required
Nginx
Reverse proxy · HTTPS termination
Certbot / Let's Encrypt
TLS certificate · free, auto-renewing · required for service workers
Pending ADRs ADR 003 (Data Platform & Analytics Pipeline), ADR 004 (Game Design), and ADR 005 (Testing Workflow) are not yet written. The IF-3 analytics pipeline destination, content schema detail, and QA process are out of scope for this document.
