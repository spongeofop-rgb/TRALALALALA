§ 01 — Context

What we are building

A Progressive Web App travel game that pulls a full game content bundle from the server at first launch, then stores everything on the player's device and runs entirely offline. The server is only contacted at install time (and optionally when content updates). All game logic, state, and progress are processed locally.

◎

Core architecture principle: Fetch once, run local

The client owns all runtime behavior after the initial bootstrap. The server's role is to deliver a well-structured content bundle and, optionally, receive analytics. No game action should require a live network connection.

§ 02 — Domains

Four domains, eight contributors

The system is divided into four domains. Each domain has a defined owner and a boundary: components within a domain are free to share internal state, but communication across domains happens only through defined interfaces (listed in §04).

Client

PWA Client

Everything that runs on the player's device — the installable shell, offline cache, local database, game logic, and all user-facing UI.

Owner: Frontend Dev · UI/UX Designers · QA · PO
Backend

Server API

The bootstrap endpoint that serves the versioned content bundle, optional session/auth, and optional event ingestion from the client.

Owner: Backend Dev
Data

Content & Data Platform

The authoritative source of game content (locations, quests, items, events), the CMS used by the game designer, and the analytics pipeline.

Owner: Data Dev · Analysts · Game Designer (content authoring)
Cross-Cutting

Shared Contracts

The bundle schema, version protocol, and analytics event schema. Agreed upon collectively; changes require sign-off from all touching domains.

Owner: PO (change control)
§ 03 — Data Flow

How data moves through the system

There are three distinct flows. Each team should understand which flows touch their domain.

§ 04 — Cross-Domain Interfaces

Where domains talk to each other

These are the only four touch-points between domains. Everything else is internal to a domain. Teams must agree on these contracts before implementation begins — they are the shared seams.

Interface From → To What crosses the boundary Owner
IF-1
Bootstrap Endpoint		 Data Platform → Backend API → Client Versioned JSON bundle: world map, locations, quests, items, events. Includes a bundle_version hash the client uses to detect staleness. Backend Dev defines schema; Data Dev populates; Frontend Dev consumes.
IF-2
Content Schema		 Data Platform → Backend API The internal DB-to-API transform. How raw CMS content maps to the bundle format. Invisible to the client. Data Dev + Backend Dev jointly.
IF-3
Analytics Events		 Client → Backend API → Data Platform Structured event payloads (session start, location visited, quest completed, etc.) fired from the client on demand. Optional; does not affect gameplay. Analysts define event schema; Backend Dev builds ingest; Frontend Dev instruments.
IF-4
Version Check		 Client → Backend API A lightweight ping on app launch: client sends current bundle_version, server responds with whether a re-pull is needed. Backend Dev + Frontend Dev.
§ 05 — Your Deliverable

What each team member submits

This is your research assignment. You are not being asked to build yet — only to scope your corner of the system so we can align before implementation.

1

Module / Service List

Name every module or service you are responsible for (minimum 3). For each: a short name, a one-sentence purpose, its inputs (what it receives or reads), and its outputs (what it produces or writes). No implementation detail required — just the contract at its boundary.

2

Relationship Diagram

A diagram showing how your modules relate to each other and how data flows between them. Any tool is acceptable (Miro, draw.io, Figma, pen and paper photographed). It must show: modules as nodes, data flows as directed edges labeled with what moves, and which interfaces from §04 your domain touches.

3

Open Questions & Assumptions

List anything you assumed that needs confirmation, and any questions that would change your scope. This prevents misalignment from compounding silently.

Scope rules — read before starting

  1. You define what your modules do, not how. Tech choices are yours to research and propose.
  2. You do not need to define the internals of another domain's modules. Reference them by the interface ID (e.g., "receives bundle via IF-1") and stop there.
  3. If a component could belong to two domains, flag it — do not silently claim it. We resolve overlaps together.
  4. The four interfaces in §04 are fixed constraints. Everything else is open.
